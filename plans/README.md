## Findings considered and rejected / deferred

- **Web UI bound to all interfaces** (`docker-compose.yml:7` `"8000:8000"`,
  no auth anywhere): anyone on the LAN can upload videos, burn CPU, and read
  outputs. Not planned because it needs a maintainer decision — if colleagues
  are meant to use the tool over LAN, binding `127.0.0.1:8000:8000` breaks
  that; if not, it's a one-line hardening. Decide, then it's an S change.
- **`_set_status` crashes if the job row was deleted mid-run**
  (`autoclip/pipeline/tasks.py:29-36`, `session.get` result used unchecked):
  only reachable by deleting rows while a job runs; the crash lands in the
  task's own except block. Too minor to plan.
- **`_ass_time` can emit `0:00:60.00`** (`autoclip/render/subtitles.py:25-30`,
  `f"{59.996:05.2f}"` rounds to `60.00`): invalid ASS timestamp on a ~0.4%
  per-event edge; libass tolerates it in practice. Fix opportunistically when
  next touching `subtitles.py`.
- **No upload size limit** (`autoclip/web/app.py:99-100` streams whatever
  arrives to disk): acceptable for a trusted internal tool; revisit only if
  LAN exposure (above) is kept.
- **Stuck non-terminal jobs after a hard worker crash** (status stays
  `rendering` forever; UI polls indefinitely): real but low-frequency on a
  single-laptop deployment, and acks_late + plan 001's visibility timeout
  means the job is eventually redelivered and re-run after worker restart.
  Revisit if the tool moves off one machine.
